# Council Review Pipeline

**Status:** Implemented â€” all 6 slices complete
**Date:** 2026-03-14 (spec), 2026-03-22 (implementation update)
**Branch:** `feature/council-review-pipeline`
**Package:** `internal/councilflow/`

## 1. Problem

The current `attest tech-spec review` performs only structural checks (required section headings present). It does not evaluate content quality, completeness, security implications, testability, or domain correctness. The technical spec is the single source of truth for everything downstream â€” execution plans, task compilation, worker instructions â€” so its quality is the ceiling for the entire pipeline.

## 2. Goal

Replace the structural-only review with a council-based review pipeline that produces a well-rounded, multi-perspective technical specification through:

1. Automated generation of review personas tailored to the spec's content and technology stack.
2. Parallel execution of persona-driven reviews using existing CLI tools.
3. A judge/editor consolidation pass that cross-validates findings and updates the spec.
4. Two fixed review rounds with full traceability of accepted and rejected findings.
5. Human final review with `@@` annotation support for remaining issues.

## 3. Flow

The number of rounds is configurable (default: 2). Each round follows the same structure:

```
PER ROUND (configurable, default 2 rounds):
  spec v(n-1) (human draft for round 1, post-consolidation for subsequent rounds)
  â†’ generate dynamic review personas (LLM-based, cached per round)
  â†’ interactive persona approval (user can remove/inspect personas)
  â†’ parallel persona reviews (staggered launch, with optional nudge pass)
      - check cache: skip reviewers whose results match current spec hash
      - run pending reviewers in parallel goroutines
  â†’ judge/editor consolidation (parallel Opus instances, one per reviewer):
      Phase 1: parallel judgment of each reviewer's findings independently
      Phase 2: apply accepted edits via string find/replace (no LLM)
      Phase 3: conflict resolution â€” re-apply failed edits against updated spec (LLM)
      - drift detection: flag removed sections, significant shrinkage (>20%)
      - empty-spec guard: abort if edits would produce an empty spec
  â†’ outputs: spec v(n), rejection-log.json, consolidation.json, per-reviewer review files
  â†’ timing: stage durations recorded for each phase

POST-PIPELINE:
  â†’ changelog.md generated (applied changes, rejections, failed edits, timing)
  â†’ final summary printed (totals, timing, file paths)

HUMAN REVIEW:
  human receives: final spec, rejection logs, all individual review outputs, changelog
  â†’ human uses @@ annotations for remaining issues
  â†’ unresolved @@ annotations block tech-spec approval
  â†’ human approves â†’ spec enters existing execution-plan pipeline
```

## 4. Review Personas

### 4.1 Fixed Personas

These run in every review regardless of the spec's content. The original spec combined security and performance into one persona; the implementation splits them into four focused personas to avoid attention-splitting:

| Persona | Backend | Focus | Perspective |
|---------|---------|-------|-------------|
| **Security Engineer** | Codex (gpt-5.4) | Threat surface, trust boundaries, authentication, authorization, input validation, secrets handling, injection vectors | "What can be exploited or tampered with?" |
| **Performance Engineer** | Gemini (gemini-3-pro-preview) | Resource bounds, scaling bottlenecks, failure under load, concurrency safety, operational cost | "What breaks at 10x scale or under resource pressure?" |
| **Testability & Test Design Reviewer** | Codex (gpt-5.4) | Acceptance criteria clarity, test boundaries, coverage gaps, deterministic verification, edge cases, regression safety | "Can every requirement be independently verified?" |
| **Architecture Reviewer** | Claude (opus) | Component boundaries, data flow, interface contracts, coupling/cohesion, consistency, extensibility | "Do the spec's abstractions hold under real-world usage?" |

Each fixed persona has exclusive review instructions that explicitly scope out other personas' responsibilities (e.g., the security reviewer is told not to flag performance issues). This reduces duplicated findings and keeps reviews focused.

### 4.2 Dynamic Personas

Generated by the judge at the start of each round based on spec analysis.

**Language/Stack Engineer** â€” The judge detects the primary language and technology stack from the spec (Go, Rust, Python, TypeScript, etc.) and generates a persona with idiomatic expertise for that stack. Examples:
- Go project â†’ Go engineer focused on concurrency patterns, interface design, error handling conventions.
- Python ML project â†’ Python engineer focused on type safety, dependency management, ML pipeline patterns.
- Rust CLI â†’ Rust engineer focused on ownership patterns, error handling with `thiserror`/`anyhow`, CLI argument design.

**Domain Expert(s)** â€” The judge identifies 1-3 domain-specific areas that require specialized review. The number and focus depend on the spec's content. Examples:
- A spec describing a file-based state machine â†’ "Distributed systems engineer" reviewing consistency and crash recovery.
- A spec describing an API â†’ "API design reviewer" checking REST conventions, versioning, backwards compatibility.
- A spec touching authentication â†’ "Identity & access management specialist" reviewing auth flows, token handling, session management.

**Constraints on dynamic personas:**
- Maximum 3 dynamic personas per round (including the language engineer).
- The judge must justify each persona choice with a one-sentence rationale tied to spec content.
- Round 2 may produce different domain personas if the spec changed significantly, but the language engineer persona stays consistent unless the stack changed.

### 4.3 Persona Definition Format

Each persona is defined as a structured config:

```json
{
  "persona_id": "go-engineer",
  "display_name": "Go Engineer",
  "type": "dynamic",
  "perspective": "Idiomatic Go expert focused on concurrency, interface design, error handling, and standard library usage.",
  "review_instructions": "Review the technical spec from the perspective of a senior Go engineer. Focus on: ...",
  "focus_sections": ["2. Architecture", "3. Canonical artifacts", "4. Interfaces"],
  "backend": "claude",
  "model_preference": "opus",
  "generated_by": "judge",
  "rationale": "Spec targets a Go CLI tool with concurrent task execution."
}
```

### 4.4 Interactive Persona Approval

Before reviews execute, the pipeline presents all personas (fixed + dynamic) for interactive approval. The user can:

- **Approve all** (`a`) â€” proceed with the full persona set.
- **Remove** (`r N`) â€” drop a persona by index (e.g., remove a redundant domain expert).
- **Detail** (`d N`) â€” inspect a persona's full config (ID, backend, model, instructions, rationale).
- **Quit** (`q`) â€” cancel the review.

This gate is skippable via `--skip-approval` or in dry-run mode.

## 5. Prompt Construction

### 5.1 Review Prompt Structure

Each reviewer receives a prompt constructed from:

1. **Persona context** â€” who they are and what perspective to take.
2. **Review mode directive** â€” tone and strictness instructions (see section 5.4).
3. **The spec** â€” either inline markdown or a file path for the reviewer to read directly. File-path mode (`--spec-path`) lets CLI tools with filesystem access perform deeper analysis.
4. **Review instructions** â€” what to look for, what output format to produce. Each persona has exclusive scoping instructions that explicitly exclude other personas' responsibilities.
5. **Codebase context** (when available) â€” relevant source files, directory structure, or Roam semantic context if the spec references existing code.
6. **Prior round context** (round 2+) â€” the previous round's findings and verdicts, so reviewers don't repeat addressed issues.

For prompts exceeding 32KB, the orchestrator pipes the prompt via stdin instead of passing it as a command-line argument, avoiding OS argument size limits.

### 5.2 The Nudge Pass

After each reviewer produces initial findings, the orchestrator can send a follow-up prompt:

> "Good findings so far. However, we expected a more thorough review from a [Persona Name]. Please review the spec again with fresh eyes. Pay particular attention to edge cases and boundary conditions you may have skimmed, implicit assumptions that are not stated explicitly, requirements that are under-specified or ambiguous, and interactions between components that could produce unexpected behavior."

The nudge output is merged with the initial findings by `finding_id` (nudge version wins for duplicates). The stricter verdict is kept.

**Note:** The nudge pass is disabled by default (`EnableNudge: false`) because it requires session resume capabilities for quality output. Enable explicitly with `--enable-nudge` when using backends that support conversation continuation.

### 5.3 Output Format

Each reviewer produces a structured findings document. Severity uses a closed set: `critical`, `significant`, `minor`. Verdict uses: `pass`, `warn`, `fail`. Confidence uses: `HIGH`, `MEDIUM`, `LOW`.

```json
{
  "persona_id": "security-engineer",
  "round": 1,
  "verdict": "warn",
  "confidence": "HIGH",
  "key_insight": "Primary concern is unauthenticated worker communication.",
  "findings": [
    {
      "finding_id": "sec-001",
      "severity": "critical",
      "category": "authentication",
      "section": "4. Interfaces",
      "location": "Section 4.2, paragraph 3",
      "description": "The spec does not specify authentication for the inter-worker communication channel.",
      "recommendation": "Add requirement: all inter-worker messages must be authenticated with HMAC-SHA256 using a per-run shared secret.",
      "fix": null,
      "why": "Without authentication, a malicious process on the same machine could inject fake completion reports.",
      "ref": null
    }
  ],
  "recommendation": "Address authentication gaps before proceeding.",
  "schema_version": 3
}
```

Output is validated at parse time: invalid verdict/confidence/severity values are rejected. Verdict is normalized to lowercase for case-insensitive acceptance.

### 5.4 Review Modes

The pipeline supports three review modes that control tone and strictness across all prompts (reviewer directives, judge directives, and persona generation):

| Mode | Flag | Reviewer Behavior | Judge Behavior |
|------|------|-------------------|----------------|
| **MVP** | `--mode mvp` | Only flag implementation blockers. Skip polish, future-phase concerns, and nice-to-haves. No `minor` severity. | Aggressively reject non-essential findings. When in doubt, reject. |
| **Standard** | (default) | Balanced review across all severity levels. Distinguish current vs. future-phase concerns. | Apply valid findings, reject future-phase issues. |
| **Production** | `--mode production` | Thorough and adversarial. Flag everything defensible. Challenge implicit assumptions. Consider 10x scale, adversarial input, 6-month maintenance. | Apply everything defensible. Only reject provably wrong findings. Require counter-evidence for rejections. |

Review mode also affects dynamic persona generation: MVP limits to 1-2 domain personas focused on buildability; Production uses all 3 slots with adversarial perspectives.

## 6. Judge/Editor Consolidation

### 6.1 Role

The judge/editor is the most critical agent in the pipeline. It must:

1. **Cross-validate** â€” Check each finding against the spec and codebase for factual accuracy. Reject findings that are wrong or based on misunderstanding.
2. **Deduplicate** â€” Multiple reviewers may flag the same issue differently. Exact duplicates from different reviewers are applied once.
3. **Apply** â€” Modify the spec to address accepted findings as minimal surgical find/replace edits. Preserve the spec's voice and structure.
4. **Explain rejections** â€” Every rejected finding gets a reason: factually incorrect, already addressed, out of scope, conflicts with a design constraint, etc. High-confidence dismissals require explicit counter-evidence and a `dismissal_rationale`.
5. **Drift-check** â€” After applying changes, verify no existing content was accidentally dropped. Flag removed sections and significant shrinkage (>20% byte reduction).
6. **Security triage** â€” Distinguish current-phase findings (single-user CLI, local execution) from future-phase findings (multi-user, network exposure). Apply current-phase findings directly; apply future-phase findings as clearly marked future requirements.

### 6.2 Parallel Judge Architecture

Rather than a single monolithic judge pass, the implementation runs **parallel Opus instances**, one per reviewer's findings. This improves throughput and keeps each judge invocation focused:

```
Phase 1: Parallel Judgment (one Opus instance per reviewer)
  â†’ Each instance judges one reviewer's findings against the full spec
  â†’ Produces accept/reject decisions with find/replace edits
  â†’ Stagger delay between launches (configurable, default 15s) to avoid rate limits
  â†’ Reviews with only minor-severity findings are skipped (not sent to judge)

Phase 2: Sequential Edit Application (no LLM, pure string ops)
  â†’ All accepted edits from all judge instances are collected
  â†’ Applied to the spec via string find/replace
  â†’ Edits with empty find/replace fields are recorded as failed
  â†’ Edits whose find text doesn't match the current spec are recorded as failed
  â†’ Empty-spec guard: abort if edits would wipe the entire spec

Phase 3: Conflict Resolution (LLM, conditional)
  â†’ Failed edits often occur because earlier edits changed the target text
  â†’ A second Opus pass receives the failed edits + the current (already-edited) spec
  â†’ Opus re-locates the closest matching text and produces updated find/replace pairs
  â†’ Edits already addressed by prior changes are rejected with explanation
  â†’ Still-failed edits after resolution are recorded in the consolidation output
```

### 6.3 Model Requirements

The judge/editor must be an Opus-class model (default: Claude Opus via `claude -p --model opus`). This role requires:
- Holding the full spec + all findings in context simultaneously.
- Making nuanced judgment calls about finding validity.
- Producing accurate spec edits without introducing new errors.
- Cross-referencing findings against each other for consistency.

The judge timeout is 30 minutes per invocation (vs. 10 minutes for reviewers), reflecting the larger context and more complex reasoning required.

### 6.4 Rejection Log Format

```json
{
  "round": 1,
  "rejections": [
    {
      "finding_id": "perf-003",
      "persona_id": "performance-engineer",
      "severity": "significant",
      "description": "The spec should mandate connection pooling for database access.",
      "rejection_reason": "The system uses file-based state (no database). The reviewer appears to have assumed a database-backed architecture.",
      "dismissal_rationale": "Spec section 3.1 explicitly states all state is persisted as JSON files under .attest/runs/",
      "cross_reference": "See spec section 3.1: 'All state is persisted as JSON files under .attest/runs/'"
    }
  ]
}
```

### 6.5 Edit Format

Each accepted edit is a surgical find/replace operation:

```json
{
  "finding_id": "sec-001",
  "persona_id": "security-engineer",
  "action": "Added authentication requirement for inter-worker messages",
  "section": "## 4. Interfaces",
  "find": "exact text to find in the spec (copy-paste from the spec)",
  "replace": "replacement text with the finding addressed"
}
```

## 7. Annotation Model

### 7.1 The `@@` Syntax

After the pipeline produces spec v2, the human reviews it. For remaining issues, the human adds inline annotations:

```markdown
## 4. Interfaces

The CLI accepts spec files via `--spec` flags.

@@ This should also support reading from stdin for piped workflows.

Tasks are dispatched via the engine's GetPendingTasks method.
```

### 7.2 Annotation Rules

- `@@` annotations are inline markers in the spec markdown.
- Each annotation is a single line starting with `@@` followed by the comment text.
- Unresolved `@@` annotations block `attest tech-spec approve`.
- Annotations are extracted and presented as review findings during approval checks.
- Once addressed (by editing the spec to resolve the concern), the `@@` line is removed.

### 7.3 Versioning and Drift Detection

Each round produces a versioned spec:
- `technical-spec-v0.md` â€” human draft
- `technical-spec-v1.md` â€” post-round-1
- `technical-spec-v2.md` â€” post-round-2

The drift detection implementation (`DetectDrift`) compares consecutive spec versions and flags:
- **Removed sections** â€” any `##` or `###` heading present in v(n-1) but absent in v(n).
- **Significant shrinkage** â€” byte count reduction exceeding 20% (e.g., "spec shrank by 35% (12000 â†’ 7800 bytes)").

Drift warnings are recorded in the `ConsolidationResult.DriftWarnings` field and printed during the review summary.

The active spec (`technical-spec.md`) is always the latest version. Previous versions are kept for audit.

## 8. Orchestration

### 8.1 Multi-Backend Support

The pipeline supports three CLI backends, each with configurable model preferences:

| Backend | Command | Default Model | Default Args |
|---------|---------|---------------|--------------|
| **Claude** | `claude` | `opus` | `-p --model opus` |
| **Codex** | `codex` | `gpt-5.4` | `exec -m gpt-5.4 -c reasoning_effort=high` |
| **Gemini** | `gemini` | `gemini-3-pro-preview` | `-m gemini-3-pro-preview -p <prompt>` |

Each persona specifies its backend and model preference. The orchestrator resolves commands to absolute paths at init time, searching `PATH`, `~/.local/bin`, `~/go/bin`, and nvm node directories to work regardless of subprocess PATH environment.

If a persona specifies a model preference, it overrides the backend's default model argument. For example, a dynamic persona could request `sonnet` instead of `opus` on the Claude backend.

### 8.2 Parallel Execution

Reviews within a round execute as parallel goroutines. The orchestrator:
1. Generates persona configs and prompts.
2. Checks for cached results (see section 8.4).
3. Launches pending reviewers in parallel goroutines with a configurable stagger delay (default: 15 seconds between launches) to avoid rate limits.
4. Waits for all to complete (per-reviewer timeout: 10 minutes).
5. Collects structured findings. Failed reviewers are logged and skipped (the round continues with available results).
6. Computes round consensus: all PASS â†’ PASS, any FAIL â†’ FAIL, mixed â†’ WARN.
7. Passes all non-trivial findings to the parallel judge instances.

Judge instances also run in parallel goroutines with the same stagger delay.

### 8.3 CLI Surface

```
attest tech-spec review <run-id>                          # full council pipeline (default: 2 rounds, standard mode)
attest tech-spec review <run-id> --rounds 3               # configurable number of rounds
attest tech-spec review <run-id> --mode mvp               # MVP mode (blockers only)
attest tech-spec review <run-id> --mode production         # production mode (thorough)
attest tech-spec review <run-id> --dry-run                 # generate personas and prompts without executing
attest tech-spec review <run-id> --force                   # re-run all reviewers (ignore cache)
attest tech-spec review <run-id> --skip-judge              # review-only mode, no consolidation
attest tech-spec review <run-id> --skip-dyn-personas       # fixed personas only
attest tech-spec review <run-id> --skip-approval           # skip interactive persona approval
attest tech-spec review <run-id> --spec-path /path/to.md   # reviewers read spec from file
attest tech-spec review <run-id> --stagger 30              # 30s between parallel launches
attest tech-spec review <run-id> --enable-nudge            # enable nudge pass per reviewer
```

The existing structural checks become a pre-flight validation before the council runs.

### 8.4 Result Caching

The pipeline caches both reviewer and judge results to enable incremental re-runs:

- **Spec hash:** A SHA-256 hash of the spec content + review mode is written to `spec-hash.txt` in each round directory. On re-run, if the hash matches and `--force` is not set, cached review files are loaded instead of re-executing.
- **Review cache:** Individual review JSON files (`review-<persona-id>.json`) are loaded if they contain a valid verdict.
- **Judge cache:** Consolidation results (`consolidation.json`) are loaded if they contain applied or rejected counts.

This means a partially-failed run can be resumed: only the reviewers/judges that failed or timed out will re-execute.

```

```

### 8.6 Changelog Generation

After all rounds complete, a `changelog.md` is generated containing:

- **Applied changes** â€” ordered by position in the updated spec (top to bottom), with finding ID, section, what changed, why (from the original finding), and severity.
- **Rejected findings** â€” with finding ID, persona, severity, the finding text, rejection reason, and counter-evidence where provided.
- **Failed edits** â€” edits that could not be applied even after conflict resolution.
- **Timing table** â€” duration of each pipeline stage (persona generation, approval, reviews, judge) and total.

### 8.7 Stage Timing

Every pipeline stage is timed and recorded in `CouncilResult.Timings`:
- `round-N/personas` â€” dynamic persona generation
- `round-N/approval` â€” interactive persona approval (only recorded if >1s)
- `round-N/reviews` â€” all reviewer executions
- `round-N/judge` â€” all judge executions + conflict resolution
- `TOTAL` â€” end-to-end pipeline duration

Timings are printed in the final summary and included in the changelog.

### 8.8 Error Handling

The pipeline uses sentinel errors for structured error classification:

| Error | Meaning | Policy |
|-------|---------|--------|
| `ErrBackendUnavailable` | CLI tool not found | Fatal |
| `ErrSubprocessFailed` | CLI process exited non-zero | Skip reviewer, continue round |
| `ErrReviewTimeout` | Reviewer exceeded timeout | Skip reviewer, continue round |
| `ErrInvalidReviewJSON` | Output couldn't be parsed as valid JSON | Skip reviewer, continue round |
| `ErrEmptyReview` | All reviewers in a round failed | Fatal for the round |
| `ErrPersonaGenerationFailed` | Dynamic persona generation failed | Fatal |
| `ErrJudgeFailed` | Judge invocation failed | Fatal for the round |
| `ErrInvalidPersonaConfig` | Duplicate persona IDs | Fatal |

## 9. Integration with Existing Pipeline

### 9.1 Gate Enforcement

The council review pipeline replaces the current structural-only review as the gate for `attest tech-spec approve`. The structural checks (required sections) become a pre-flight that runs before the council.

```
attest tech-spec draft    â†’ records spec v0
attest tech-spec review   â†’ runs council pipeline, produces spec v2 + review artifacts
attest tech-spec approve  â†’ validates: review passed, no unresolved @@ annotations, spec hash matches
```

### 9.2 Downstream Impact

No changes to the downstream pipeline. The execution plan draft still reads from the approved `technical-spec.md`. The difference is that the approved spec is now substantively reviewed, not just structurally checked.
